# UX-01 — Cierre UX de la web del MVP

**Teralya · Versión 1.0 · 17/07/2026 · Estado: APROBADO POR EL CTO; DIRECCIÓN VISUAL AUTORIZADA POR EL CEO EL 18/07/2026**

## Objetivo

Registrar las decisiones UX ya implementadas para completar la web sin crear nuevas pantallas ni ampliar el MVP.

## Principios obligatorios

- Una pantalla corresponde a un código PT existente.
- La información comercial prioriza claridad, confianza y operación.
- Los estados Vacío, Cargando, Correcto y Error deben ser comprensibles.
- Las acciones irreversibles o económicas requieren confirmación explícita.
- Los roles conservan navegación y layouts diferenciados.
- La autorización real pertenece al backend.
- El sitio debe ser operable en escritorio y móvil y mediante teclado.

## Dirección visual premium autorizada

- Teralya adopta una identidad visual oscura, editorial y contemporánea, con acentos oro, magenta, violeta y azul.
- El logotipo principal usa el isotipo de cuatro prismas de color y el nombre `TERALYA` en mayúsculas.
- La tipografía editorial se reserva para titulares; navegación, formularios y datos usan una tipografía sans legible.
- La portada combina origen, experiencia doméstica y confianza mediante fotografía de viñedo, entrega en casa y consumo responsable.
- Se permite una pieza audiovisual breve de una bodega ficticia cuando se identifique expresamente como historia visual o contenido editorial, sin atribuirla a una bodega real.
- Catálogo, ficha de vino, ficha de bodega, checkout y áreas privadas comparten cabecera, pie, tarjetas, bordes, espaciado y contraste.
- Los activos externos deben disponer de licencia compatible y no pueden introducir marcas, testimonios o características comerciales no aprobadas.
- El rediseño es exclusivamente visual: reutiliza rutas, APIs, formularios, roles y flujos existentes.

## Salvaguarda de alcance visual

Los mockups de referencia no autorizan funcionalidades adicionales. No se implementan club, puntos, favoritos, reseñas, colecciones comerciales nuevas, newsletter, PayPal, Apple Pay, Google Pay, devoluciones garantizadas, compensación de carbono ni atención personalizada salvo que exista aprobación funcional y contractual posterior.

## Cancelación de Pedido en PT-COM-007

- La acción se ubica después del detalle económico y de direcciones.
- Se usa el texto directo **Cancelar pedido** y el encabezado **Cancelar contrato**.
- Se exige una casilla de confirmación y una segunda confirmación del navegador.
- La interfaz no afirma que el Pedido está cancelado hasta recibir el estado del backend.
- Se distinguen visual y textualmente `procesando`, `completada` y `fallida`.
- Cuando la cancelación directa ya no está disponible se enlaza la política de desistimiento.

## Condiciones de envío

- La edición vive dentro de PT-BOD-002, en un `fieldset` propio.
- Se explica el formato de países mediante códigos ISO separados por comas.
- La ficha pública de Bodega agrupa destinos, plazos, costes, transportista, restricciones y empaquetado.
- La ficha de vino enlaza a la sección de condiciones de su Bodega.
- La ausencia de un dato no genera afirmaciones inventadas ni valores ficticios.

## Accesibilidad mínima verificada

- Etiquetas asociadas a todos los controles.
- Confirmación obligatoria accesible mediante teclado.
- Mensajes de estado con `aria-live` cuando procede.
- Un único contenido principal por recorrido.
- Enlaces y botones con propósito explícito.
- Cabeceras y jerarquía semántica conservadas.

## Fuera de alcance

No se añaden favoritos, reseñas, chat, CRM, club, subastas, aplicación móvil, IA, traducción automática ni soporte avanzado.
